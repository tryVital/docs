title: "Proxy Ordering Registrable Testkits"
description: "Recommended flows for proxy ordering and inventory workflows with registrable testkits."
---

Junction's registrable testkit flow supports ordering testkits before the final patient registration step is completed. These are also known as "unregistered" testkits because the patient registration step happens after the kit is ordered. However, each testkit order must still be associated with a Junction user at the time the order is created.

Because of this, the `user_id` used when creating the order should be treated as the user who will ultimately own the order and results in Junction.

This workflow guide outlines recommended implementation patterns for two flows where the patient may not be the person or system initiating the order:

- A user orders a registrable testkit on behalf of another person.
- A clinic orders registrable testkits in bulk for on-hand inventory.

These are general recommendations. Your implementation may need to adjust these patterns based on your specific user model, inventory workflow, registration workflow, and result display/retrieval requirements.

<Note>
  The Junction `user_id` used when creating a testkit order should represent the eventual patient and result owner for that specific kit. However, the person who pays for, initiates, or physically receives the kit does not necessarily need to be the same as the Junction user associated with the order.
</Note>

## Relevant API requests

These flows use the following Junction API requests:

- Use [`POST /v2/user`](/api-reference/user/create-user) to create a Junction user when the intended patient or inventory user does not already exist in Junction.
- Use [`GET /v2/user/{user_id}`](/api-reference/user/get-user) when you already have the Junction `user_id` and need to retrieve that user.
- Use [`GET /v2/user/resolve/{client_user_id}`](/api-reference/user/resolve-user) when you have your own `client_user_id` and need to look up the corresponding Junction user.
- Use [`POST /v3/order/testkit`](/api-reference/lab-testing/create-unregistered-order) to create a registrable testkit order.
- Use [`POST /v3/order/testkit/register`](/api-reference/lab-testing/register-order) to register the kit when the final patient information is available.

<Note>
  In the flows below, "create or resolve a user" means: first look up the existing Junction user using `GET /v2/user/{user_id}` if you already store the Junction `user_id`, or `GET /v2/user/resolve/{client_user_id}` if you store your own user identifier. If no Junction user exists for that intended patient or inventory kit, create one with `POST /v2/user`.
</Note>

## Scenario 1: User A orders a testkit for User B

### Example

User A places an unregistered testkit order for User B. User B is the intended patient and will register the kit using their own demographic information.

### Recommended flow

1. User A starts the order in your application.
2. Your application determines that the intended patient and result owner is User B.
3. Create or resolve User B in Junction. If User B already exists and you store their Junction `user_id`, retrieve them with [`GET /v2/user/{user_id}`](/api-reference/user/get-user). If you store your own `client_user_id`, retrieve them with [`GET /v2/user/resolve/{client_user_id}`](/api-reference/user/resolve-user). If they do not exist in Junction, create them with [`POST /v2/user`](/api-reference/user/create-user).
4. Place the registrable testkit order with [`POST /v3/order/testkit`](/api-reference/lab-testing/create-unregistered-order) using User B's Junction `user_id`.
5. User B registers the kit with [`POST /v3/order/testkit/register`](/api-reference/lab-testing/register-order).
6. Order and results are associated with User B.

### Important detail

Junction does not model this as a separate proxy-ordering concept. From Junction's perspective, this should look like a normal [registrable testkit order](/lab/workflow/order-registrable-testkit) for User B.

The distinction between User A and User B should be handled by your application before calling Junction. The Junction order should be created using the `user_id` for the intended patient, not the person who initiated the order in your application.

User A can remain the purchaser, account holder, or initiating actor in your own system, but Junction should receive User B's `user_id`.

### If User B already exists

If User B already exists in your system and already has a Junction user ID, reuse User B's existing Junction `user_id`. You can retrieve that user with [`GET /v2/user/{user_id}`](/api-reference/user/get-user) if you store the Junction user ID, or [`GET /v2/user/resolve/{client_user_id}`](/api-reference/user/resolve-user) if you store your own user ID.

If User B exists in your system but does not yet have a Junction user, create one with [`POST /v2/user`](/api-reference/user/create-user) before placing the order.

<Warning>
  Do not place the order using User A's Junction `user_id` if User B is the person who will register the kit. At registration, the patient demographics submitted for User B are applied to the Junction user associated with the original order. In this case, User A's stored demographics would be overwritten with User B's registration details.
</Warning>

## Scenario 2: Bulk registrable kits for inventory

### Example

Your clinic wants to order 100 registrable testkits and keep the physical kits in inventory or on site before the final patients are known. For example, you may need kits available at a location so they can be handed to patients later.

### Recommended flow

For inventory workflows, create a dedicated Junction user for each kit or order. Throughout the rest of this document, these users will be referred to as "inventory users."

For example, if you need 100 kits:

1. Create 100 Junction users with [`POST /v2/user`](/api-reference/user/create-user) to represent inventory-held kits. Each inventory user should have a distinct `client_user_id` in your system, such as an inventory item ID or kit allocation ID.
2. Place 100 registrable testkit orders with [`POST /v3/order/testkit`](/api-reference/lab-testing/create-unregistered-order), one per inventory user.
3. Store a mapping in your system between each physical kit and its Junction user/order.
4. When a final patient is assigned to a kit or claims a kit, link that person in your system to the corresponding Junction user/order.
5. Register the kit with [`POST /v3/order/testkit/register`](/api-reference/lab-testing/register-order) using the Junction user/order already associated with that inventory kit.

### Why one Junction user per kit

Each registrable testkit order is tied to the Junction user supplied at order creation. When the kit is registered, the registration information is applied to that order's existing Junction user. Because of this, each inventory kit should have its own Junction user.

<Warning>
  Do not reuse one Junction inventory user across multiple kits.
</Warning>

## Suggested inventory mapping

Your system should maintain a mapping between your inventory records, your patients, and Junction identifiers.

| Field | Description |
| --- | --- |
| Internal inventory ID | Your internal ID for the physical kit |
| Inventory `client_user_id` | Your stable identifier for the inventory user created in Junction |
| Junction user ID | The Junction user created for this inventory kit |
| Junction order ID | The Junction order created for this kit |
| Sample ID / kit ID | The identifier used to register or identify the kit |
| Patient ID | Your patient ID, once the kit is assigned or claimed |
| Status | Available, claimed, registered, completed, expired, etc. |

### Example mapping

| Internal Inventory ID | Inventory `client_user_id` | Junction User ID | Junction Order ID | Sample ID | Patient ID | Status |
| --- | --- | --- | --- | --- | --- | --- |
| kit_001 | inventory_kit_001 | junction_user_001 | order_001 | sample_001 | null | available |
| kit_002 | inventory_kit_002 | junction_user_002 | order_002 | sample_002 | patient_123 | registered |
| kit_003 | inventory_kit_003 | junction_user_003 | order_003 | sample_003 | patient_456 | claimed |

## Inventory claim and registration flow

1. Identify the kit using your internal inventory ID, barcode, sample ID, or other kit identifier.
2. Look up the corresponding Junction user and Junction order in your system. If you need to re-fetch the Junction user, use [`GET /v2/user/{user_id}`](/api-reference/user/get-user) or [`GET /v2/user/resolve/{client_user_id}`](/api-reference/user/resolve-user), depending on which identifier your system stores.
3. Link the patient record in your system to that Junction user/order.
4. Complete the kit registration flow using [`POST /v3/order/testkit/register`](/api-reference/lab-testing/register-order) for that Junction user/order.
5. Store the relationship so future result lookup works correctly.

## Patients with multiple kits

A patient may be assigned multiple kits over time.

In this inventory model, that patient may be linked to multiple Junction users because each pre-ordered inventory kit already has its own Junction user. That is expected.

Your system should not assume:

**one patient = one Junction user**

For inventory flows, your system should support:

**one patient = one or more Junction users/orders**

### Example

The same patient is assigned three kits over time:

| Patient ID | Junction User ID | Junction Order ID |
| --- | --- | --- |
| patient_123 | junction_user_001 | order_001 |
| patient_123 | junction_user_045 | order_045 |
| patient_123 | junction_user_089 | order_089 |

When showing results to this patient, your application should look up all Junction users/orders mapped to that patient and retrieve the relevant results from those mappings.
